What is User Lifecycle Management?

User lifecycle management is a comprehensive process that involves managing a user's identity and access throughout their association with an organization.

The User Lifecycle Management process outlines the steps involved in user registration, access provisioning, access deprovisioning, and governance. It serves as a reference for admins, super admins and other business personas to efficiently manage user roles and permissions.

User Lifecycle Management Personas

User Lifecycle Management is managed by different user personas, each with specific permissions:

Organization owner

Organization owners are responsible for managing organizational-level configurations. Whichever application is a part of that organization, will have those groups also applied to it. Organization owners are on a higher level than Applications Owner. Organization owners can create organization level groups, add or remove applications from the organizations which App Owners or App admins are unable to do. Organization Owners have full access to all applications and users. They have the ability to create, modify, and remove users and roles. Organization owners can manage policies and governance.

Application Owner

An application owner is responsible for a specific application. They can manage users and access within that application. They are assigned by an application admin and can also assign others as application owners. Application owners can create groups, assign users, and define roles. However, they cannot view other applications.

Group Owner

A group owner has restricted access within the application system. They can manage user access within a specific group but cannot change all configurations. Group owners can manage users within their assigned group or division, but they cannot access users outside their division.

User Administrator

A user administrator manages user-related operations, such as user registration, adding or removing users, assigning roles, and handling security concerns like deactivating compromised accounts, ensures users have the correct permissions based on governance policies. They can search for users via email or username and manage their access within the system.

User Registration

This tab is used for onboarding new users. User invitation is the process by which an individual creates an account within an application or system. This typically involves providing personal information, such as an email address and password, and completing any required verification steps. Once registered, the user gains access to the application's features and functionalities (only when the user is invited through the permission Assign access to this invitation).

  4. Choose the Application Redirect URL from the dropdown.
  5. Click Assign access to this invitation.
  6. Steps to follow in the Associate Application Group:
    1. Click Associate Application Group.
    2. Search the group name, and click Add. (Only one group can be associated with an invitation.)
    3. Click Confirm Assignment.

  7. Create Group User Attribute:

    Notes:

    • Attribute Name is a user-defined identifier (key) for additional metadata attached to a user's access profile. (Example: Attribute Name: Role, Attribute Value: Healthcare Admin OR Attribute Name: Persona, Attribute Value: Super Administrator)
    • Attribute Value is the specific data assigned to the attribute name, used for policy enforcement or decision-making.
    1. Create Group User Attribute

Access Granting and Role Assignment

Role Assignment is the process of linking predefined roles (bundles of permissions) to users or groups. Roles define:

  • What a user can access (e.g., applications, data).
  • How they can interact (e.g., view, edit, approve).

The two types of Role Assignment are:

  • User based role assignment
  • Group based role assignment

User based Role Assignment

This step involves assigning the attribute necessary permissions to users. Users are assigned to roles and groups based on their department or application.

Steps

  1. Navigate to the Users section.
  2. Select Registered Users in User Type.
  3. Select Email Address / Username / UUID from the Search dropdown.
  4. Type Email Address / Username / UUID of the user as per the selection. (Example: ssmith@guerrillamail.com /Ssmith/0c474546-0b8d-406f-9734-c122bbea03f1)
  5. Click Search.
  6. Click on the name of the user.
  7. Select the Access tab and click Assign Roles.

    Screenshot: User Profile View with Access Tab

  8. Select an application from the dropdown.
  9. Select application group from the dropdown.
  10. Click Add > Confirm Assignment.

    Pop-up: Assign Roles

  11. Users will successfully be assigned with roles.

Group Based Role Assignment

Group-based Role Assignment automatically grants access permissions to users by assigning them to predefined groups (e.g., departments, teams, or divisions). Roles are attached at the group level instead of individually.

Steps

  1. Go to the Access section within the application to view the user's current access.
  2. If no access is displayed, select the application for which you want to view or assign roles. (Example: Type and select JK Healthcare to view roles for that application.)
  3. The system will display the roles and groups assigned to the user for the selected application. (Example: The user might be an admin for "JK Healthcare".)
  4. To assign new roles, click on Assign Roles, select the application, choose the appropriate group (Group-based access ensures users inherit permissions automatically), and click Add to assign.
  5. Verify Role Assignment by searching for the user; the assigned role will be visible in the user's profile.

Advanced Search and Bulk Operations

  • Advanced Search Option: If you don't have the user's email address, username, or OHID, use the advanced search option. Search using the first name and last name. (Example: If user’s name is Steve Smith, then enter the first name Steve and last name Smith to find the user.)
  • Bulk Import for Access Granting: Use the Import User functionality. Prepare an Excel file with user details in the predefined format (Example: OHID, username, and roles). Upload the file to import users and assign access. New users will receive an email invitation; existing users will have access assigned directly.

Fast-Track Registration vs. Direct Self-Registration

Fast-Track User Registration with Access: Ensures users are productive immediately after registration by granting access during the onboarding process. An admin invites a user, optionally assigning access (roles/groups) upfront. Upon clicking the link, the user registers and is automatically granted pre-assigned access and redirected to the application. No separate access-grant step is needed.

Direct Self-Registration Process: Allows users to register independently (e.g., via a portal), but requires manual access assignment afterward by an admin via the Admin Console. No access is granted automatically upon signup.

Role Inheritance and Access Policies

Role Inheritance

Role Inheritance refers to the automatic assignment of roles (and their associated permissions) to users based on their membership in specific groups or divisions. This eliminates manual assignments and ensures consistent access control across teams.

Example: If a user is added to the "Pharmacy" group, they inherit the "Pharmacy Manager" role automatically, along with permissions like "View Patient Records" or "Process Orders." If a user is removed from the group, the role (and permissions) is automatically revoked.

Access Policies

Access Policies are rules that govern how roles and permissions are assigned, enforced, and managed. These policies define:

  1. Inheritance Rules: Whether roles are auto assigned via group membership (for new applications) or manually assigned (for legacy systems).
  2. Conditions: Criteria for granting/revoking access (e.g., job function, location).
  3. Compliance: Requirements like periodic access reviews or separation of duties.
  4. Custom Attributes: Metadata (e.g., "Department=Cardiology") used to refine access decisions.

In essence, role inheritance streamlines access management, while access policies enforce security, compliance, and operational efficiency.

Compliance and Access Policies

Access Governance Framework

The Access Governance Framework in the Admin Console IGA solution is designed to enforce secure, scalable, and self-service access management across applications and user roles.

Core Components of the Framework

  • Persona-Based Access Control: Supports five key personas: Super Admin, Application Admin, Group Owner, User Admin, and Org Owner, each with defined privileges.
  • Role & Permission Modeling: Roles (e.g., "Pharmacy Manager") are created with attached permissions (actions like "view" or "edit").
  • Group-Based Access: Users inherit roles via group membership (e.g., "Pharmacy" group members auto get "Pharmacy_Admin" role). Supports auto-inheritance (new apps) or manual assignment (legacy apps).
  • Custom Attributes: Additional metadata (e.g., "Department=Cardiology") can be assigned to users/groups for fine-grained policy enforcement.

Governance Mechanisms

Mechanisms include Self-Service Access Assignment, Auto Revocation, Bulk Import/Export, and Application Lifecycle Control.

Policy-Based Role Management

Objective: Streamline role assignments using structured policies, adhering to governance principles like Least Privilege and Separation of Duties.

  1. Role Creation: Navigate to Applications > Roles in Identity Console to define roles and descriptions.
  2. Permission Assignment: Under each role, attach predefined permissions (e.g., "Manage Users").
  3. Role-to-Group Mapping: Assign roles to groups (e.g., "Pharmacy" group gets "Pharmacy Manager" role).

Audit and Reporting Features

Objectives: Provides visibility into user access and enables compliance audits. Includes:

  • Audit Logs: Records user access grants/revocations, role/group modifications, and application lifecycle changes.
  • Reporting: Provides User Access Summary, Group Compliance, and Application Status.
  • Periodic Recertification: Schedule reviews for role assignments and group memberships.

Group-Based Access Control (GBAC)

Objective: Simplify access management through groups. Access is controlled through groups rather than direct assignments, reducing governance complexity.

  1. Group Creation: Define groups (e.g., "OPD," "Cardiology") with descriptions and custom attributes.
  2. Role Inheritance: Assign roles to groups. Users added to the group inherit roles automatically (if auto-inheritance is enabled).
  3. User Assignment: Add users via Bulk Import (CSV) or Manual Entry.
  4. Access Revocation: Remove users from groups to revoke inherited roles.

Access Revocation and Governance

Access revocation is the process of removing or restricting a user's access to applications, roles and permissions within an organization. Governance ensures that only necessary access is granted when a user leaves or changes roles.

Method 1: Group-Based Access Revocation

Group Owner or Super Admin is responsible for this action.

  1. Applications > [Select Application] > Access > Groups > [Select Group]. (Example: For the JKAdmin group under JK Healthcare application.)
  2. Locate User in the Group: Under the group’s Users tab, search for the target user (e.g., Steve Smith).
  3. Revoke Group Membership: Click Remove next to the user’s name. All roles auto inherited via the group are revoked immediately. (Example: Revoking Steve Smith from the JKAdmin group removes his Doctor role.)

Method 2: Manual Role Revocation

Application Owner or Super Admin are responsible for this action, typically for legacy applications with no auto-inheritance.

  1. Users > Search User > Select [username]> Access.
  2. Search for User and Application: Enter the username (e.g., "Steve Smith") and select the application (e.g., "UHCP Report").
  3. Revoke Assigned Roles: Under the Roles section, click Revoke next to the role to remove.

Governance Policies

Governance policies in the Admin Console’s IGA system are rule-based frameworks that enforce secure, compliant, and standardized access controls across an organization. These policies automate decision-making for access provisioning, revocation, and role assignments while ensuring alignment with regulatory requirements (e.g., HIPAA, SOX).

Policy 1: Auto-Inheritance for New Applications

Super Admins, Application owners and Organization owners are responsible to perform this action.

  1. Applications > [New Application] > Settings > Group Policy.
  2. Toggle Auto-Inheritance to On. Result: Users added to groups automatically inherit roles assigned to those groups.

Policy 2: Custom Attributes for Enhanced Control

Application owners and Organization owners are responsible to perform this action.

  1. Applications > [Application] > Custom Attributes Add metadata (e.g., Department=Cardiology, Location=NYC).
  2. Apply Attributes to Users/Groups: Use attributes to enforce policies (e.g., restrict access to users with Location=NYC).

Policy 3: Application Lifecycle Management

Super Admin or Application Owner can manage the Application Status.

  1. Applications > [Name of the Application] > Status.
  2. Set to Inactive to block all authentication requests (e.g., during a security breach).

User Access Management

User Access Management is the end-to-end process of provisioning, modifying, and revoking user access to applications and resources within the IGA (Identity Governance and Administration) framework. It combines self-service capabilities and role-based controls to ensure secure and efficient access lifecycle management.

Core Components of User Access Management

  • Personas & Roles: Includes five key personas (Super Admin, Application Admin, Group Owner, User Admin, Org Owner) and Roles/Permissions that bundle access rights.

Table: Access Assignment Methods

Method Process Use Case
Manual Assignment Admins assign roles to users via UI. Ad-hoc access needs.
Bulk Import (CSV) Upload predefined roles/groups for mass onboarding. Large-scale hires (e.g., 10K users).
Fast-Track Registration Invite users with pre-assigned roles; access granted upon registration. Critical roles (e.g., healthcare providers).
Group Inheritance Users auto-inherit roles when added to a group. Team/department-based access (e.g., "Finance" group).

Access Revocation

  • Group-Level: Remove user → All group roles revoked instantly.
  • Role-Level: Selective revocation of individual roles.
  • Bulk Revocation: Revoke all roles for a user/group via CSV.

Use Cases

Practical examples of ULM in action:

  • Hospital Management System: Doctors, nurses, and billing staff have separate access levels. Only authorized users can enter restricted areas like the operation theater.
  • Enterprise Application Access: Employees are assigned to departments with predefined access. Access is revoked automatically when an employee leaves.
  • Healthcare Provider Onboarding: New providers receive access on day one for immediate productivity.
